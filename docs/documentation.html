Overview

Solar Neo provides a single toolchain for curating Fedora (DNF5) packages and Flatpak remotes. Whether you manage a homelab or a campus of desktops, the CLI keeps dependency state and release metadata synchronized across every workstation.

The documentation below highlights the most common workflows. For release specific change logs, head over to the version archive.

Installation

Download the newest bundle from the latest release page and follow the included installation guide. The archive contains the installer script and usage notes.

If you need a previous release, pick the appropriate build from the version archive.

Core commands

The v0.7 CLI ships with a single entry point named solar. Run python3 -m solarneo if your shell does not expose the launcher on the PATH. Each command prints a banner before executing and returns a non-zero exit code when something goes wrong, making it easy to script.

  • solar version — show the installed client build, query the update API for the newest tag, and offer to install it in-place.
  • solar update — skip the version comparison prompt and immediately download and install the newest GitHub release after confirmation.
  • solar sys — list the package backends detected on the host (DNF5 and/or Flatpak).
  • solar search <query> — proxy the query to both DNF5 and Flatpak and print any matches that come back.
  • solar install <name> — install a package via DNF5 when available, falling back to Flatpak.
  • solar remove <name> — uninstall the package through the same backend that handled the install.
  • solar notes <app> — fetch community and developer release notes for the given application.
  • solar notes <app> add "message" — submit a new community note to the shared feed.
  • solar devnote add <app> "message" — publish a developer note when the SOLAR_DEV_TOKEN environment variable is set.

If you prefer unattended upgrades, pipe a default answer into the prompts, for example yes | solar update.

Configuration

The client keeps all state inside your home directory so that system access is not required.

  • ~/.local/share/solarneo/solarneo — the installed client code. Backups live alongside it under ~/.local/share/solarneo/.backup.
  • ~/.config/solarneo/ — store any optional configuration files or deployment manifests you need.
  • ~/.cache/solarneo/ — download cache used during updates.

When you invoke solar update the CLI automatically snapshots the current install into the backup folder before replacing it with the newly downloaded build.

Automation tips

Solar Neo is script-friendly and can be embedded in provisioning pipelines.

  • Combine solar search with traditional UNIX tools like grep to filter DNF5 or Flatpak results for automation.
  • Pipe yes into solar update or solar install when you need unattended confirmations.
  • Export SOLAR_DEV_TOKEN before running solar devnote add … so that release engineers can post maintainer notes from CI.
Tip: Script solar version during health checks to quickly learn which nodes are out of date before kicking off a fleet-wide update.

Troubleshooting

  • Download failures — re-run solar update. The CLI prints the HTTP error it encounters when it cannot reach GitHub releases.
  • Unzip errors — verify the release archive downloaded completely. Partial downloads will surface as unzip failed messages.
  • No backend detected — install either dnf5 or flatpak so that package operations have a provider.

Still stuck? File an issue on GitHub with the full console output from the command you ran.
